## 3. Ambiguity Findings

### A1 — Tool is read-only but `ReadOnlyHint` is missing
**Location:** `vfx-list-particles` — `Tool_VFX.ListParticleSystems.cs:26`
**Issue:** This tool only reads scene state and never mutates anything, but the `[McpTool(...)]` attribute does not set `ReadOnlyHint = true`. Compare `component-list` (`Tool_Component.List.cs:26`), which correctly sets the hint. Without this hint, downstream UIs that filter by safety class will treat this as a write-capable tool, defeating the purpose of the hint.
**Evidence:** `[McpTool("vfx-list-particles", Title = "VFX / List Particle Systems")]` — no `ReadOnlyHint` parameter.
**Confidence:** high

### A2 — Description does not enumerate the full output schema
**Location:** `vfx-list-particles` — `Tool_VFX.ListParticleSystems.cs:27`
**Issue:** The method-level `[Description]` lists "max particles, duration, loop, emission rate, and simulation space" (5 fields) but the actual output reports 11 fields including `playOnAwake`, `startLifetime`, `startSpeed`, `startSize`, `emission.enabled`, and `isPlaying`. An LLM reading only the description would not know it can also retrieve the play state or the start-* curves, and may invoke a hypothetical second tool when this one already covers the case.
**Evidence:** Description (line 27): `"Lists all ParticleSystem components in the current scene with key settings including max particles, duration, loop, emission rate, and simulation space."` vs. actual output fields on lines 47–57.
**Confidence:** high

### A3 — Description does not disambiguate from `component-list`
**Location:** `vfx-list-particles` — `Tool_VFX.ListParticleSystems.cs:27`
**Issue:** No "use this when X, not Y" clause. An LLM tasked with "show me particle systems in the scene" must choose between this tool and `component-list` (which can also enumerate ParticleSystem components if pointed at each GameObject). The description never explains that this tool is the scene-wide aggregator and that `component-list` is per-GameObject.
**Evidence:** See A2 — the description is purely declarative.
**Confidence:** medium

## 5. Capability Gaps

This is the dominant section for this domain. The VFX directory contains exactly one read-only tool and zero authoring tools. Standard Unity VFX workflows are largely uncovered.

### G1 — No way to create, configure, or remove a Particle System
**Workflow:** A Unity dev wants to "add a hit-spark particle effect to the Player prefab." Steps a human takes: GameObject menu → Effects → Particle System; tweak main module (loop, duration, start lifetime/speed/size); tweak emission rate and bursts; assign material; save prefab.
**Current coverage:** `component-add` (Component domain) can attach a `ParticleSystem` component to an existing GameObject. There is no VFX-domain tool, and no other domain tool, that configures any ParticleSystem module after creation.
**Missing:** Tools for `ParticleSystem.MainModule` (loop, duration, simulationSpace, startLifetime, startSpeed, startSize, startColor), `EmissionModule` (rateOverTime, bursts), `ShapeModule` (shape, radius, angle), `ColorOverLifetimeModule`, `SizeOverLifetimeModule`, `RendererModule` (material, sortingLayer). Underlying API: assignment to module structs which Unity treats as live-write proxies (e.g. `var m = ps.main; m.loop = true;`).
**Evidence:** `Tool_VFX.ListParticleSystems.cs` is the only file in the domain. Grep for `ParticleSystem` across `Editor/Tools/` matches only this file.
**Confidence:** high

### G2 — No playback control (Play / Pause / Stop / Simulate)
**Workflow:** During iteration, a dev wants to trigger an effect from the editor without entering Play Mode, or simulate it at time T to inspect it visually.
**Current coverage:** None.
**Missing:** Wrappers for `ParticleSystem.Play()`, `Pause()`, `Stop(includeChildren, ParticleSystemStopBehavior)`, `Simulate(time, withChildren, restart)`, `Clear()`. The current tool reports `isPlaying` but provides no way to change it.
**Evidence:** `Tool_VFX.ListParticleSystems.cs` line 57 reads `ps.isPlaying` but no Play/Stop/Simulate API surface is wrapped anywhere in `Editor/Tools/`.
**Confidence:** high

### G3 — `MinMaxCurve.constant` accessor is correct only for `ParticleSystemCurveMode.Constant`
**Workflow:** Inspect a ParticleSystem whose `startLifetime` is set to `Random Between Two Constants` or `Curve` mode (a common authoring pattern).
**Current coverage:** `vfx-list-particles` always prints `main.startLifetime.constant`, `main.startSpeed.constant`, `main.startSize.constant`, `emission.rateOverTime.constant`.
**Missing:** Mode-aware reporting. When `MinMaxCurve.mode != ParticleSystemCurveMode.Constant`, `.constant` returns the lower bound only and silently misrepresents the actual curve. The output should branch on `mode` and report `constantMin`/`constantMax`, `curve`, or `curveMultiplier` accordingly.
**Evidence:** `Tool_VFX.ListParticleSystems.cs` lines 52–56 unconditionally read `.constant`. No check of `main.startLifetime.mode` or equivalent.
**Confidence:** high

### G4 — No support for the modern VFX Graph (`UnityEngine.VFX.VisualEffect`)
**Workflow:** In URP/HDRP projects (this project is a 2D URP roguelike per CLAUDE.md), VFX Graph is the modern authoring system. Devs expect to list, play, pause, and set exposed properties on a `VisualEffect` component.
**Current coverage:** None. Grep for `VisualEffect` across `Editor/Tools/` returns zero matches.
**Missing:** Wrappers for `VisualEffect.Play()`, `Stop()`, `Pause()`, `SendEvent(string)`, `SetFloat`/`SetInt`/`SetVector2`/`SetVector3`/`SetTexture`/`SetGameObject` and the `HasFloat` etc. probe methods. Also a list/inspection tool analogous to `vfx-list-particles` but for VFX Graph instances.
**Evidence:** `Grep "VisualEffect" Editor/Tools/` → 0 files.
**Confidence:** high

### G5 — Read-only inspection cannot target a single GameObject or filter by name
**Workflow:** A dev (or LLM) has just configured `Player/HitSpark` and wants to verify only that one system, not dump every particle system in the scene.
**Current coverage:** `vfx-list-particles` always returns all ParticleSystem components scene-wide.
**Missing:** Optional parameters such as `gameObjectId` (entity ID), `nameFilter` (glob/regex), or `includeInactive` (bool) to scope the report. Without these, large scenes produce noisy output and force the LLM to grep through prose.
**Evidence:** `ListParticleSystems()` signature on line 28 takes no parameters; line 32 always calls `Object.FindObjectsByType<ParticleSystem>(FindObjectsSortMode.None)` with no filtering.
**Confidence:** high

### G6 — No tool for `ParticleSystem.EmitParams` / one-shot bursts
**Workflow:** A dev wants to fire a single burst of N particles at a given position for testing or for a script-driven event.
**Current coverage:** None.
**Missing:** A wrapper around `ParticleSystem.Emit(EmitParams, count)` or `Emit(count)`.
**Evidence:** Same as G2 — no playback / emission control surface exists.
**Confidence:** medium (less critical than G1–G4, but a common dev request)

## 7. Notes

**Cross-domain dependencies noticed:**
- VFX leans on `component-add` (Component domain) for any kind of ParticleSystem instantiation. If VFX grows authoring tools (G1), Ramon should decide whether `vfx-create-particle-system` should exist as a high-level wrapper, or whether VFX stays inspection/control-only and continues to delegate creation to `component-add`. The Animation domain made the high-level-wrapper choice (e.g. `animation-configure-controller`) and that pattern reads well.

**Strategic question for the reviewer:**
- Should the VFX domain expand to cover VFX Graph (G4)? This is a real architectural call. VFX Graph is package-gated (`com.unity.visualeffectgraph`), so its tools may need a `#if VFX_GRAPH_AVAILABLE` guard or a separate optional assembly. Worth discussing before the consolidation-planner is invoked.

**Workflows intentionally deferred:**
- I did not enumerate every sub-module of `ParticleSystem` (Trails, Noise, Texture Sheet Animation, Lights, Sub-Emitters, Collision, Triggers, External Forces, Custom Data). G1 captures the high-level gap; the planner should decide module-by-module coverage based on Jurassic Survivors's actual usage.

**Open questions for the reviewer:**
1. Is `vfx-list-particles` actively used by the Jurassic Survivors test project, or is it dead code from an early prototype? The answer changes how aggressive the consolidation can be.
2. If the answer to G4 is "yes, cover VFX Graph", does Ramon want VFX Graph and ParticleSystem unified under one `vfx-*` namespace, or split as `vfx-particle-*` vs. `vfx-graph-*`?

**Limits of this audit:**
- The audit covers a single file (1/1, balanced). No part of the report is downgraded for incomplete coverage.
- Absence claims about VFX Graph ("no `VisualEffect` coverage anywhere") were verified by a project-wide `Grep` for `VisualEffect` returning zero matches — a complete-coverage check.
